Flask + Jinja2でVWAPバックテスト比較Webアプリを作って。

【構成】
- app.py（Flaskメイン）
- fetch_data.py（J-Quants APIデータ取得）
- strategies/vwap.py
- strategies/volume_price.py（出来高加重価格戦略）
- strategies/ma_cross.py（移動平均クロス戦略）
- templates/index.html（トップ）
- templates/result.html（比較結果）
- templates/detail.html（銘柄×戦略詳細）
- static/style.css
- Dockerfile
- docker-compose.yml
- .dockerignore
- .env（JQUANTS_API_KEY, FLASK_SECRET_KEY）

- .gitignore（.envを除外）
- requirements.txt

【データ】
- J-Quants V2 API Free Plan を使用する
  - エンドポイント: GET https://api.jquants.com/v2/equities/bars/daily
  - 認証ヘッダー: x-api-key: <APIキー>（Bearer方式やリフレッシュトークンは使わない）
  - レスポンス: { "data": [...], "pagination_key": ... }
  - 日足のカラム名: Date, Code, O/H/L/C, Vo(出来高), AdjO/AdjH/AdjL/AdjC/AdjVo（調整後）
  - Freeプランはデータが約12週間遅延する（例: 今日が2026-04-18なら最新データは2026-01-24前後）
  - 直近12週分のデータを取るには、fromをそれより前に設定してAPIが返す範囲の末尾12週を採用する
  - **ページネーション必須**: レスポンスの `pagination_key` が空でない限り、同パラメータに
    `pagination_key` を付け直して次ページを取得するループを実装する（12週でも複数ページになることがある）
  - **購読期間外 400 の自動リトライ**: Freeプランは購読期間が過去範囲に限定され、範囲外の日付を
    指定すると 400 が返る。エラー本文には `subscription covers the following dates: YYYY-MM-DD ~ YYYY-MM-DD`
    形式で許容範囲が含まれるため、正規表現で終了日をパースして from/to を再計算し、一度だけ自動リトライする
    - 判明した購読終了日はプロセス内にキャッシュして以降のリクエストで再利用する（毎回 400 を踏まない）
    - 内部伝播用の専用例外（例: `_SubscriptionRangeError`）を用意し、400 受信時は即 raise してループ外でリトライする
- 初期資金：100万円（銘柄ごとに独立）

【銘柄選択UI】
- index.htmlで銘柄を選択できるUI
- デフォルトは1銘柄（7203）のみ（Freeプランのレート制限が厳しいため）
- +ボタンで銘柄入力欄を追加（最大10銘柄）
- -ボタンで銘柄入力欄を削除（最低1銘柄）
- 銘柄コードは **4桁英数字**（例: 7203 / 285A キオクシア）。近年は末尾に英字が入る新規上場銘柄が増えているため数字縛りにしない
  - HTML の pattern は `[0-9A-Za-z]{4}`、JS で大文字に自動正規化する
  - バックエンドでは `code.isalnum() and len(code)==4` で検証し、大文字化してから API / キャッシュキーに使う
- 選択した銘柄リストをPOSTでFlaskに送信してバックテスト実行

【戦略詳細】
1. VWAP戦略
   - 終値 > VWAP → 翌日買い
   - 終値 < VWAP → 翌日売り（手仕舞い）
   - **VWAP の計算粒度**: 取得した12週の期間全体の累積 VWAP（Σ(典型価格×出来高) / Σ出来高）を使う
     （日次リセットの intraday VWAP ではなく、期間累積の running VWAP）
     - 典型価格 = (High + Low + Close) / 3

2. 出来高加重価格戦略
   - 出来高が過去5日平均の1.5倍超 かつ 終値が前日比プラス → 翌日買い
   - 出来高が過去5日平均の1.5倍超 かつ 終値が前日比マイナス → 翌日売り（手仕舞い）

3. 移動平均クロス戦略
   - 短期MA(5日)が長期MA(25日)を上抜け → 翌日買い
   - 短期MA(5日)が長期MA(25日)を下抜け → 翌日売り（手仕舞い）

【リスク管理オプション（全戦略共通）】
- 任意で損切り/利確の自動手仕舞いを有効化できる（index.html のチェックボックス）
  - 損切り: エントリ価格から指定% 下落で手仕舞い（デフォルト 3%）
  - 利確: エントリ価格から指定% 上昇で手仕舞い（デフォルト 5%）
- 約定ロジック（_engine.py）:
  - エントリ翌日以降の各バーで判定（同日内のエントリ+即時手仕舞いは許容しない）
  - ギャップダウンで Open < 損切り価格 → Open 価格で約定（現実的な最悪ケース）
  - ギャップアップで Open > 利確価格 → Open 価格で約定（ギャップ益を反映）
  - 同日内に両方トリガー可能なら **損切りを優先**（保守側）
- トレード履歴には `reason` フィールドを持たせる（SIGNAL/STOP/TP）
  - detail.html のメインチャートでは reason ごとにマーカーを色分け:
    - 売り約定（シグナル）: 赤 ▼ triangle-down
    - 損切り約定: 濃茶 × x
    - 利確約定: 青 ★ star
  - トレード履歴テーブルに「理由」列を追加してバッジ表示
- result.html / detail.html の subtitle に「損切り -3% / 利確 +5%」のような設定表示

【画面構成】
1. index.html
   - 銘柄選択UI（上記参照）
   - 「バックテスト実行」ボタン

2. result.html
   - 選択銘柄×3戦略のヒートマップ表（最終損益を色で表現、プラス緑・マイナス赤）
     - セル内に「損益/勝率/取引回数」を併記
   - 各セルをクリックで詳細ページへ

3. detail.html
   - 選択した銘柄×戦略の詳細
   - 以下の2グラフをPlotlyで表示：
     1. ローソク足 + インジケータ（戦略別）を重ねたメインチャート
        - VWAP戦略: 期間累積 VWAP を1本の線で重ねる
        - 移動平均クロス戦略: **MA5（短期）と MA25（長期）の2本を別の色で同時に重ねる**
          （クロスが視認できるよう両方とも必須。片方だけでは戦略の挙動が読めない）
        - 出来高加重価格戦略: 5日移動平均出来高の1.5倍閾値を比較するため、参考として
          終値や移動平均価格などの補助線を1本重ねてもよい（必須ではない）
        - ▲/▼ マーカーは「実際に約定したトレード」のみ表示し、トレード履歴テーブルの行と1対1で対応させる
          （生のシグナル列を描くと毎日 ▲▼ が付き、ポジションを既に持っている日の
           "買えなかったシグナル" まで見えてしまい、履歴テーブルと不整合になるため）
        - マーカー位置は約定価格、ホバーで「BUY/SELL 株数・価格・損益」を表示
     2. 損益曲線（折れ線）
   - トレード履歴テーブル（日付・売買・価格・損益・累計損益）
   - ※ 出来高バーチャートは削除済み（VWAP は指標線に、MAクロスは無関係、
     出来高加重は閾値線なしで視覚的意味が薄いため）

【ルーティングとPOST/Redirect/GET】
- バックテストの送信は POST /run で受け、結果はプロセス内キャッシュに保存してから
  GET /result へ 302 リダイレクトする（PRG パターン）
- GET /result がセッション経由でキャッシュを読んで result.html を描画する
- detail.html の「結果一覧に戻る」リンクは url_for('show_result') で GET /result を指す
  （javascript:history.back() は POST /run に戻ろうとして「フォーム再送信の確認」が出るため使わない）
- キャッシュが失効/未保持の場合は / にリダイレクト

【グラフ】
- Plotlyを使用（matplotlibは不使用）
- FlaskからJSON形式でデータを渡しJinja2テンプレートで描画
- 全グラフはズーム・ホバー・凡例トグル対応
- result.html は Plotly ヒートマップのみ（比較バーチャートは削除済み）

【環境変数】
- python-dotenvで.envから読み込む
- 読み込む変数：
  - JQUANTS_API_KEY（J-Quants V2 APIキー。ダッシュボードで発行）
  - FLASK_SECRET_KEY（セッション暗号化用）
- V2 APIはAPIキー方式のためメール/パスワードやリフレッシュトークン交換は不要
- リクエストヘッダーは x-api-key: <APIキー> を直接付与する（Authorization Bearer は使わない）
- APIキーに有効期限はない（再発行・削除は可能）
- API呼び出し時に401/403が返ったら認証エラー扱い（キャッシュなし）
- .envはgitignoreで除外済み
- .env.exampleの中身：
  JQUANTS_API_KEY=your_api_key_here
  FLASK_SECRET_KEY=local-development-secret
- 旧名 JQUANTS_REFRESH_TOKEN を設定している既存ユーザー向けにフォールバックで読み込んでもよい

【エラー表示】
- エラーはHTML上にわかりやすく表示する
- 特にJ-Quantsの認証エラー（APIキー無効/未設定/ダミー値）は専用ブロックで強調表示する
  - 発生した原因（「APIキーが無効」「未設定」「ダミー値のまま」など）
  - HTTPステータスコードとJ-Quants APIのレスポンス本文（<details>で折りたたみ可能）
  - 対処手順（ダッシュボードでAPIキー発行 → .env更新 → docker compose up -d --force-recreate web）
  - restart ではなく force-recreate が必要である旨を明記する（env_file は作成時のみ読まれるため）
- レート制限（HTTP 429）は専用ブロックで表示する
  - 実装上は専用例外（例: JQuantsRateLimitError）に retry_after 属性を持たせ、
    Retry-After ヘッダ（秒数/HTTP-date両対応）をパースして表示する
  - 429 はまず指数バックオフで最大3回自動リトライし、それでも駄目なら専用エラー画面へ
  - 画面では「しばらく待って再実行」「銘柄数を減らす」などの対処を案内する
  - Freeプランは 429 後に約5分遮断されるため、画面に「再試行可能時刻」の目安（現在時刻+5分）を表示する
  - 既に取得済みでプロセス内キャッシュに残っている銘柄コードも画面に併記し、
    「この銘柄はAPIを消費せず再取得できる」ことをユーザーに示す
- 認証エラー/レート制限エラーを検知した時点で全銘柄のバックテストを打ち切り、無駄なリトライはしない
- 実装上は認証エラー用の例外クラス（例: JQuantsAuthError）を用意し、
  detail属性にAPIレスポンス本文を保持してテンプレートに渡す
- 銘柄ごとの個別エラー（データなし、購読期間外、上場廃止等）は箇条書きで一覧表示する

【レート制限への対処】
- J-Quants のプラン別レート制限（参考値）:
  - Free: 5 リクエスト/分（超過で約5分ブロック）
  - Light: 60 リクエスト/分
  - Standard: 120 リクエスト/分
  - Premium: 500 リクエスト/分
- Freeプランでも使い物になるよう、以下のレート制限緩和策を実装する:
  1. 日足データをプロセス内キャッシュに TTL 1時間で保持する
     （同一銘柄を再取得する際にAPIを消費しない）
  2. 銘柄ごとのAPI呼び出しの間に 2秒のウェイトを入れる
  3. デフォルト銘柄数は1のみ（ユーザーが明示的に増やすまで安全側）
  4. キャッシュ有効銘柄を返す public ヘルパ（例: cached_codes()）を用意し、
     429 発生時に「キャッシュ済み銘柄」として画面に表示する

【ローディング表示】
- バックテスト実行中はオーバーレイ + スピナーを表示する
- 初期状態（ページロード直後）は必ず非表示で、フォーム送信時のみJSで表示する
- 実装上の注意:
  - HTMLの hidden 属性は User Agent Stylesheet の display:none で効くが、
    同じ要素に CSS クラスで display:flex 等を指定すると詳細度で負けて常時表示になる
  - そのため CSS 側で `.loading-overlay[hidden] { display: none; }` を明示し、
    hidden 属性を付けた時に確実に非表示になるようにする
  - 結果ページに遷移すれば DOM 自体が入れ替わるので自動で消える
- 動作確認項目:
  - 初期表示でスピナーが出ていないこと
  - 送信後は応答が返るまでスピナーが出続け、次ページで消えていること
  - HTML5バリデーションに失敗した場合はスピナーを出さずに入力エラー表示のみ

【スマホ対応（max-width: 640px）】
- タップ領域: ボタン/入力欄は最小 44px（Apple HIG / Material 準拠）
  - `.btn { min-height:44px }`、`.btn-icon { 44x44 }`、`.code-input { min-height:44px }`
  - 入力欄の font-size は 16px 固定（iOS Safari の自動ズーム防止）
- 結果テーブル: `.table-wrap` に overflow-x:auto を前提に、各セルへ min-width と white-space:nowrap を付けて横スクロールに確実にフォールバック
  - 1列目（銘柄）は position:sticky で左端に固定し、横スクロール中も見失わないようにする
- Plotly ヒートマップ: window.innerWidth < 640 のとき JS で font.size=10、margin 縮小、height を「80px × 行数 + 120」に増やして文字重なりを回避
- Plotly ローソク足/損益曲線: 同条件で height・margin・font.size を縮小、凡例を下部横並び（legend.orientation='h'）に変更
- ローソク足は Python 側で rangeslider を無効化済み（xaxis_rangeslider_visible=False）
